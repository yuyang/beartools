# Gmail Fetch 设计文档

## 目标

为 beartools 增加 `gmail fetch` 子命令，抓取最近 `n` 天内带 `INBOX` 标签的 Gmail 邮件，使用现有 LLM 能力生成本批邮件的统一摘要，并输出模型判断的最重要 10 封邮件与总体概览。结果需要同时打印到终端，并按抓取时间保存到 `./email/` 目录。

## 范围

本次只实现以下能力：

- 新增 `beartools gmail fetch` 命令。
- 使用 Gmail API 查询最近 `n` 天且带 `INBOX` 标签的邮件。
- 首次运行使用本地浏览器 OAuth 授权，并在本地保存 token 文件。
- 读取邮件主题、发件人、时间和正文文本，作为 LLM 输入。
- 让模型对整批邮件统一判断，输出最重要 10 封邮件摘要和总体概览。
- 当查询结果超过 100 封时，明确提示“邮件过多，仅处理前 100 封”，并只处理前 100 封邮件。
- 将结果保存到 `./email/YYYY-MM-DD_HH-MM-SS.md`，同时输出到终端。

不包含以下能力：

- 不处理广告邮件、存档、删除、打标签、已读未读状态修改。
- 不下载附件。
- 不做每封邮件单独生成完整摘要文件。
- 不做 HTML 富格式保留，仅提取文本内容。

## 用户交互

命令形态：

```bash
beartools gmail fetch
```

建议支持参数：

- `--days`：抓取最近多少天，提供默认值，避免每次手动输入。
- `--max-results`：本地处理上限，默认固定为 `100`。

命令执行流程：

1. 读取 Gmail 配置。
2. 检查本地 token 是否存在且可用。
3. 若 token 不可用，发起本地浏览器 OAuth 授权。
4. 调用 Gmail API 查询 `INBOX` 最近 `n` 天邮件。
5. 若邮件数量超过 100，先提示用户本次只处理前 100 封。
6. 拉取待处理邮件详情并提取正文文本。
7. 调用现有 LLM 生成统一摘要。
8. 将结果输出到终端并保存到 `./email/` 目录。

## 配置设计

在现有 `Config` 中新增 Gmail 配置段，建议字段如下：

- `gmail.client_secret_file`：Google OAuth client secret 文件路径。
- `gmail.token_file`：本地 OAuth token 文件路径。
- `gmail.output_dir`：输出目录，默认 `email`。

client secret 文件通常放在本地，不进入仓库；token 文件也应放在本地配置目录或项目私有目录中，并保持忽略提交。

## 模块划分

### CLI 层

- `src/beartools/cli.py`
  - 注册 `gmail` 命令组。
- `src/beartools/commands/gmail/command.py`
  - 暴露 `fetch` 子命令。
  - 负责命令参数解析、异常展示、终端打印。

### Gmail 业务层

- `src/beartools/gmail.py`
  - 封装 OAuth 获取客户端。
  - 封装 Gmail 查询与邮件详情读取。
  - 封装正文提取、结果组装、Markdown 输出。
  - 封装调用 LLM 生成统一摘要。

建议在该模块中定义明确的数据结构，例如：

- `GmailMessageSummaryInput`：单封邮件输入数据，包含主题、发件人、时间、正文文本等。
- `GmailFetchResult`：本次抓取结果，包含抓到的数量、实际处理数量、是否截断、输出文件路径、最终摘要文本等。

### LLM 摘要层

复用现有 `LLFactory` 和 runtime 机制，不新增独立模型配置。

摘要 prompt 关注以下约束：

- 输入是一批邮件，不是单封邮件任务。
- 输出先给“最重要的 10 封邮件”。
- 每封只保留简短摘要，突出为什么重要。
- 再给“总体概览”，总结主要主题、待处理事项、潜在风险。
- 如果模型发现多封邮件属于同一主题，可在总体概览中聚合。

## Gmail 查询设计

使用 Gmail API 的查询语法进行预过滤：

```text
label:inbox newer_than:{days}d
```

查询原则：

- 只拿 `INBOX`，不处理其他系统标签。
- 先拉 message id 列表。
- 再逐封拉详情，提取 headers 与 payload。

正文提取策略：

1. 优先读取 `text/plain`。
2. 如果没有纯文本，则从 `text/html` 转成纯文本。
3. 控制单封邮件正文长度，避免 prompt 过长。

## 100 封上限规则

本次需求要求：如果超过 100 封，需要提示，邮件多只处理 100 封。

因此规则固定为：

- Gmail API 查询结果如果大于 100 封：
  - 终端提示“本次命中 X 封邮件，超过处理上限 100，仅处理前 100 封”。
  - 输出 Markdown 中也写明该提示。
  - 实际传给模型的只有前 100 封。
- 若不超过 100 封，则全部处理。

这里的“前 100 封”按 Gmail 查询返回顺序处理，通常对应最近邮件优先。

## 输出设计

输出目录固定为 `./email/`，不存在则自动创建。

文件名格式：

```text
email/YYYY-MM-DD_HH-MM-SS.md
```

Markdown 结构建议如下：

```markdown
# Gmail 邮件摘要

- 抓取时间：2026-05-01 10:30:00
- 查询范围：最近 3 天
- 命中邮件数：126
- 实际处理数：100
- 说明：超过处理上限，仅处理前 100 封

## 最重要的 10 封邮件

1. [主题] ...
   - 发件人：...
   - 时间：...
   - 摘要：...

## 总体概览

...
```

终端输出与 Markdown 内容保持一致，允许在终端中省略过长正文，只展示最终摘要内容。

## 错误处理

需要明确处理以下错误：

- Gmail client secret 文件不存在或配置缺失。
- OAuth 失败、本地授权端口不可用、token 刷新失败。
- Gmail API 请求失败。
- 邮件正文为空或解析失败。
- LLM 不可用或摘要生成失败。
- 输出目录创建失败、文件写入失败。

命令层需要把这些错误转成用户可读的中文提示，并以非零退出码结束。

## 测试设计

测试覆盖分为三层：

### 1. CLI 测试

- `gmail fetch` 参数解析正确。
- 成功时会输出摘要路径和关键信息。
- 邮件超过 100 封时会输出截断提示。
- 失败时会输出中文错误提示并返回非零退出码。

### 2. Gmail 业务测试

- 正确构造 `label:inbox newer_than:Nd` 查询。
- 超过 100 封时只处理前 100 封。
- 正文优先读取 `text/plain`，缺失时降级 `text/html`。
- 输出目录不存在时自动创建。
- Markdown 文件名按抓取时间生成。

### 3. LLM 摘要测试

- 不直接依赖真实外部模型，使用替身验证 prompt 结构和结果拼装。
- 校验输出中包含“最重要的 10 封邮件”和“总体概览”。

## 与现有代码的契合点

- 命令注册方式对齐 `fetch`、`record`、`siyuan` 等现有模式。
- 配置读取方式复用 `config.py` 的 dataclass + Dynaconf 解析模式。
- 摘要模型复用现有 `LLFactory`，避免重复引入新的模型抽象。
- 测试风格对齐 `tests/test_fetch.py` 的命令测试和 mock 风格。

## 风险与约束

- Gmail OAuth 依赖本地浏览器授权，首次执行环境必须具备交互能力。
- 邮件正文内容可能很长，因此必须做截断，否则容易超出模型上下文。
- Gmail 返回的 MIME 结构可能复杂，正文提取需要优先保证稳定而不是完美还原格式。
- LLM 生成“最重要的 10 封邮件”带有模型判断成分，结果不是绝对规则，但符合本次需求。

## 结论

采用 Gmail API + 本地 OAuth + 现有 LLM 的方案，在当前代码结构中新增 `gmail fetch` 命令，抓取 `INBOX` 最近 `n` 天邮件，并在超过 100 封时提示且仅处理前 100 封，最终输出“最重要的 10 封邮件”和“总体概览”，同时在输出内容中明确包含本次处理的邮件个数和抓取的天数，并保存到 `./email/` 目录。
