# LLM 运行时节点优先级设计

## 背景

当前 `LLRuntime` 在初始化活动节点和节点失效切换时，使用 `random.choice(...)` 从健康节点池中随机选取节点。

这与当前配置结构的语义不一致：

- `agent.primary` 应表示默认优先使用的主节点
- `agent.candidates` 应表示按顺序排列的候选节点

现状会导致：

- 启动后不一定优先选中 `primary`
- 候选节点之间也不按配置顺序生效
- 实际运行行为不可预测，和配置阅读直觉不一致

## 目标

将 LLM 节点选择策略改为**严格顺序优先**：

1. 只要 `primary` 健康，就始终默认选择 `primary`
2. 当 `primary` 不可用时，按 `candidates` 在配置中的顺序，选择排在最前面的健康节点
3. 当前活动节点失效后，切换到剩余健康节点中顺位最高的节点
4. 不引入随机策略，不做加权选择，不做透明重试

## 非目标

本次不处理以下内容：

- 不新增负载均衡能力
- 不实现节点权重配置
- 不实现请求级自动重放
- 不改动节点健康探测逻辑
- 不改动现有配置结构

## 设计方案

### 1. 保持节点收集顺序

保留当前 `_collect_configured_nodes()` 的顺序：

```python
[agent_config.primary, *agent_config.candidates]
```

这样 `healthy_nodes` 的顺序天然表达优先级：

- 下标 0：`primary`
- 后续下标：按配置顺序排列的 `candidates`

### 2. 活动节点选择改为顺序优先

将 `LLRuntime._choose_active_fingerprint()` 从“随机选一个可用节点”改为“返回当前可用节点列表中的第一个”。

选择规则：

1. 遍历 `healthy_nodes`
2. 跳过已失效节点（`_failed_fingerprints`）
3. 跳过 `exclude_fingerprint` 指定节点
4. 返回第一个满足条件的节点指纹
5. 若没有可用节点，则返回 `None`

这保证：

- 初始化时默认选 `primary`
- 切换时总是落到顺位最高的备节点

### 3. 失效切换行为

`mark_node_failed()` 的整体行为保持不变：

- 只影响后续默认节点选择
- 不对当前失败请求做透明重放

仅变更切换目标：

- 如果失效的是当前活动节点，则切换到剩余可用节点中的第一个
- 如果失效的不是当前活动节点，仅记录其失效状态，不改变当前活动节点

### 4. 日志行为

已增加的 `LLFactory.create()` 日志无需改动。

在顺序优先策略下，这条日志会自然反映：

- 默认命中 `primary`
- `primary` 失效后命中排位最前的 candidate

## 测试设计

采用 TDD，先补失败测试，再改实现。

### 需要覆盖的行为

1. **初始化默认选主节点**
   - 给定 `[primary, candidate-1, candidate-2]`
   - 初始化后活动节点应为 `primary`

2. **主节点失效后按顺序切到第一个候选节点**
   - 当前活动节点为 `primary`
   - `primary` 失效后，应切到 `candidate-1`

3. **前置候选节点失效后继续按顺序切换**
   - `candidate-1` 已失效时
   - 下一次切换应选择 `candidate-2`

4. **非活动节点失效不应影响当前活动节点**
   - 当前活动节点仍健康
   - 标记其他候选节点失效后，不应切换活动节点

## 风险与兼容性

### 风险

1. **行为变化是显式的**
   - 之前依赖随机分流的行为将消失
   - 但从配置命名语义看，这属于修正而不是破坏

2. **流量更集中在前序节点**
   - 所有请求默认优先打到 `primary`
   - 这与“主节点 / 候选节点”的配置语义一致

### 兼容性结论

这是一次行为收敛：从“随机平权”收敛到“主节点优先、候选顺序回退”。

对当前配置结构、调用方接口、日志格式均保持兼容。

## 实施后预期结果

在配置如下时：

1. `primary`
2. `candidate-1`
3. `candidate-2`

预期行为为：

- 默认使用 `primary`
- `primary` 失效后使用 `candidate-1`
- `candidate-1` 也失效后使用 `candidate-2`
- 不再出现随机命中后序节点的情况
