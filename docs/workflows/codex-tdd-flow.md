# Codex TDD 工作流

把 Codex 会话约束成适合 side project 的 `plan -> test -> implement -> verify -> review -> fix -> docs-sync` 循环。

## 触发语
- `实现功能：<需求>`
- `按商业开发流程做：<需求>`
- `实现功能：<需求>，可以用 subagent`

只有用户明确说“可以用 subagent”或等价表述时，才允许分派 subagent。主线程始终负责节奏控制、最终集成、验证和汇总；subagent 只承担可并行、边界清晰且不阻塞主线程的探索、测试建议或 diff review。

## 阶段门禁
- 每个阶段结束后，汇报产出、验证结果、风险和下一步。
- Planner 是硬门禁，不是建议流程：未通过 Planner Exit Checklist 前，必须停在 Planner，不得进入 Test Writer、Executor，也不得补写测试或生产代码。
- Test Writer -> Executor -> Verify -> Reviewer 默认自动推进，由 Codex 按计划、测试结果、实现完成度和风险判断。
- Reviewer 结论、阻塞性修复范围、最终 Documentation Sync 结果需要用户明确同意。
- 用户不同意、发现计划/测试/实现/Verify 标准需要变化时，回到对应阶段更新产出并重新确认。

## 1. Planner
目标：澄清需求、读取相关上下文、完成 brainstorm 和遗漏检查，形成最小可交付计划。

硬要求：
- 只允许新增或更新 `docs/plans/<YYYY-MM-DD>-<short-slug>.md`，不改生产代码和测试代码。
- 按需读取 `docs/codemap.md`，并检索 `docs/plans/` 中相关历史计划，确认设计边界、风险、偏离记录和验证经验。
- 先轻量 brainstorm 2-3 个方向，列出取舍和易遗漏风险，再收敛到推荐方案。
- 必须明确使用 `grill-me` skill 做 Grill Gate：逐个追问关键分支、依赖决策和潜在遗漏；每次只问一个问题，并给出推荐答案。
- Grill Gate 中能通过读代码回答的问题先自行探索；关键阻塞点解决后及时收敛。
- 明确目标、非目标、影响模块/文件/测试范围、业务规则、架构选择、数据库变更等需确认点。
- 必须定义三类 Verify 标准，并得到用户明确同意：
  - 自动化验证：最小验证命令、完整验证命令和预期结果，通常包含本需求相关的 `pytest`、`ruff`、`mypy`。
  - 冒烟端到端验证：由用户给出少数关键路径测试，说明入口、输入、预期结果，以及外部服务/凭据/人工环境依赖。
  - 全面端到端验证：由用户给出覆盖主要分支的多个测试；可省略，但必须得到用户明确确认，并记录省略原因和风险。
- 有必须确认的问题时先问用户；计划未被明确同意前，不进入后续阶段。

Planner Exit Checklist，全部满足才能离开 Planner：
- 已写入计划文档 `docs/plans/<YYYY-MM-DD>-<short-slug>.md`。
- 计划文档已包含必要上下文、推荐方案、影响范围、TDD/测试策略、三类 Verify 标准、风险和需确认问题。
- 已明确使用 `grill-me` skill 完成 Grill Gate，且计划文档记录问题、推荐答案和遗漏检查结论。
- 用户已明确确认计划正确。
- 用户已明确确认自动化验证、冒烟端到端验证、全面端到端验证三类 Verify 标准；若省略全面端到端验证，用户已明确确认省略原因和风险。
- 不存在未回答的业务规则、架构选择、数据库变更或其他阻塞问题。

计划文档必须包含：背景和目标、历史上下文、非目标、Brainstorm 选项和推荐方案、`grill-me` / Grill Gate 问题与遗漏检查结论、影响范围、TDD/测试策略、三类 Verify 标准、分步实施计划、风险/回滚/需确认问题。

可选 subagent：`explorer` 探索影响范围、现有模式、相关测试；`default` 辅助 brainstorm 或边界分析，只输出建议，不修改文件。

## 2. Test Writer
目标：先写或调整测试，尽量形成 TDD 红灯。

要求：
- 优先修改 `tests/`；不改生产代码，除非测试辅助结构需要极小调整且已说明原因。
- 写完后运行最小测试命令，确认新测试暴露当前缺口。
- 无法做红灯测试时说明原因，改用回归测试或 characterization test。
- 测试足以约束计划范围时自动进入 Executor；若暴露计划错误、Verify 标准变化或范围外实现需求，回到 Planner。

可选 subagent：`worker` 在清晰限定的测试文件中补测试。

## 3. Executor
目标：按计划实现最小必要变更。

要求：
- 只改与需求直接相关的文件，不做无关重构。
- 遵守项目规范：Python 3.13+、完整类型提示、中文注释/文档字符串、依赖精确版本。
- 每完成一层关键变更后，按风险决定是否先运行局部测试。
- 实现覆盖计划范围后自动进入 Verify；若必须改变计划、测试策略或 Verify 标准，回到对应阶段确认。

可选 subagent：`worker` 仅在写入范围互不重叠时承担独立实现切片。

## 4. Verify
目标：按 Planner 定义的 Verify 标准证明改动可工作。

默认命令：
```bash
uv run pytest tests/ -xvs
uv run ruff check .
uv run mypy .
```

要求：
- 先跑自动化最小验证，再按风险跑完整验证。
- 执行 Planner 中的冒烟端到端验证；如要求全面端到端验证，也执行并记录结果；如已确认省略，记录原因和剩余风险。
- 验证不足时补充行为验收或针对性测试，并按门禁回到 Planner 更新 Verify 标准。
- 环境、依赖、外部服务不可用导致失败时，说明原因和替代证据。
- 不为了通过测试删除或弱化已有测试。
- 证据满足 Verify 标准后自动进入 Reviewer；失败或证据不足时进入 Fix Loop 或回到对应阶段。

## 5. Reviewer
目标：审查当前 diff 的 bug、回归风险和测试缺口，并合并轻量 audit。

要求：
- 不修改文件。
- 使用 `docs/checklists/review.md`，必要时参考 `docs/checklists/audit.md` 的安全、数据、发布风险项。
- findings 按严重程度排序，引用文件和行号；无问题时说明剩余风险或测试缺口。
- side project 默认不单独执行 Auditor；只有涉及安全、持久化数据、外部发布、敏感配置或用户明确要求时才单独 audit。
- Reviewer 结束后等待用户同意；如有阻塞问题，进入 Fix Loop 前确认修复范围。

可选 subagent：`reviewer` 或 `explorer` 只读 diff 和上下文，输出 findings。

## 6. Fix Loop
目标：处理 Verify/Review 反馈。

要求：区分必须修复、后续优化和需用户决策的问题；由 executor 或主线程修复；修复后重新运行相关验证和 review。3 轮以内可自动继续；连续 3 轮以上仍未解决同类阻塞问题时，暂停并让用户决定是否继续、调范围或接受风险。

## 7. Documentation Sync
目标：同步对后续 agent 有价值的稳定知识和最终计划记录。

要求：
- 检查是否需要更新 `AGENTS.md`、`docs/codemap.md` 和本次计划文档。
- 计划文档必须更新到与最终实现一致：实际改动、最终 Verify 结果、偏离原因、未完成项和后续建议。
- 只记录稳定事实，例如新模块职责、入口命令、关键数据流、测试命令、项目约束变化；不记录一次性调试、临时计划、过期 TODO 或执行日志。
- 无需更新时在最终回复说明原因；文档同步涉及格式或链接风险时做轻量检查。
- Documentation Sync 结束后，等待用户同意最终文档同步结果，再输出完成结论。

## 最终输出
保持简短，包含：改了什么、验证了什么、文档同步情况、未运行或失败的命令、剩余需用户决策的风险。
