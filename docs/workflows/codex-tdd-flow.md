# Codex TDD 工作流

本流程用于把 Codex 会话约束成适合 side project 的 `plan -> test -> implement -> verify -> review -> fix -> docs-sync` 循环。

## 触发语
用户可以用以下方式触发：
- `实现功能：<需求>`
- `按商业开发流程做：<需求>`
- `实现功能：<需求>，可以用 subagent`

只有当用户明确说"可以用 subagent"或等价表述时，才允许 Codex 分派 subagent。

## 总体职责
- 主线程负责节奏控制、最终集成、测试验证和结果汇总。
- subagent 只承担可并行、边界清晰的任务，例如影响范围探索、测试建议、diff review。
- 不把正在阻塞主线程的关键任务外包给 subagent。
- 不重复执行 subagent 已经完成且可信的工作。

## 阶段门禁
- 每个执行阶段结束后，必须向用户汇报本阶段产出、验证结果、风险和建议的下一步。
- Planner 阶段完成后，必须得到用户对计划文档和 Verify 标准的明确同意，才能进入 Test Writer 或 Executor。
- Test Writer -> Executor、Executor -> Verify、Verify -> Reviewer 默认不需要用户再次同意；Codex 应根据计划、测试结果、实现完成度和风险自行判断是否继续推进。
- Reviewer 结论、阻塞性修复范围和最终 Documentation Sync 结果仍需要用户明确同意。
- 如果用户明确不同意当前阶段结果，或提出需要调整的点，必须继续迭代当前阶段，直到问题解决或用户明确同意为止。
- 如果执行中发现计划、测试、实现或验证标准需要变化，必须先回到对应阶段更新产出，并重新取得用户明确同意。
- 未得到 Planner 明确同意前，不得为了推进流程而提前修改 Test Writer 或 Executor 负责的文件。

## 执行阶段
### 1. Planner
目标：澄清需求、保留必要的 brainstorm、读取相关代码、形成最小可交付计划。

要求：
- Planner 阶段只允许新增或更新计划文档，不修改生产代码和测试代码。
- 计划文档必须写入 `docs/plans/<YYYY-MM-DD>-<short-slug>.md`。
- 开始制定新计划前，按需读取 `docs/codemap.md`，并检索 `docs/plans/` 下的历史计划，查找相关设计、实现边界、风险、偏离记录和验证经验。
- 先进行轻量 brainstorm：提出 2-3 个可行方向、关键取舍和可能被忽略的产品/工程风险。
- 对复杂、模糊或高风险需求启用 Grill Gate：像 `grill-me` 一样逐个追问计划/设计中的关键分支，每次只问一个问题，并给出推荐答案。
- Grill Gate 中如果问题可以通过阅读代码回答，先探索代码，不把可自行确认的问题抛给用户。
- Grill Gate 只持续到关键阻塞点被解决；不要为了追问而追问。
- brainstorm 后必须收敛到一个推荐方案；不能停留在发散讨论。
- 明确目标和非目标。
- 列出预计影响的模块、文件和测试范围。
- 明确三类 Verify 标准：自动化验证、冒烟端到端验证、全面端到端验证。
- 自动化验证必须包含本次需求相关的 `pytest`、`ruff`、`mypy` 或其他可自动运行命令，并区分最小验证命令和完整验证命令。
- 冒烟端到端验证必须由用户给出少数关键路径测试，用于证明需求在真实入口上可工作；如果依赖外部服务、凭据或人工环境，需要提前标出。
- 全面端到端验证必须由用户给出多个覆盖主要分支的测试；全面端到端验证可以省略，但必须在 Planner 阶段得到用户明确确认，并在计划文档中记录省略原因和风险。
- 标出业务规则、架构选择、数据库变更等需要用户确认的点。
- 若存在必须确认的问题，先停下来问用户。
- 计划文档写完后，必须等待用户确认计划正确，才能进入 Test Writer 或 Executor 阶段。
- 必须确定用户已经给出或明确同意三类 Verify 标准，且对全面端到端验证是执行还是省略作出明确确认，才能进入 Test Writer 或 Executor 阶段。
- 用户确认后，后续阶段以该计划文档为执行基准；如果执行中发现计划错误，先更新计划并再次确认。
- 如果用户没有明确同意计划，继续迭代 Planner 阶段，不进入测试或实现。

计划文档必须包含：
- 背景和目标。
- 历史上下文：引用相关 `docs/codemap.md` 条目和 `docs/plans/` 历史计划，并说明可复用结论或需要避开的风险。
- 非目标。
- Brainstorm 选项和推荐方案。
- Grill Gate 问题、推荐答案和用户确认结论。
- 影响范围。
- TDD/测试策略。
- Verify 标准：
  - 自动化验证：最小验证命令、完整验证命令和预期结果。
  - 冒烟端到端验证：用户给出的少数关键路径测试、执行入口和预期结果。
  - 全面端到端验证：用户给出的多个主要分支测试、执行入口和预期结果；若省略，记录用户明确确认、省略原因和风险。
- 分步实施计划。
- 风险、回滚和需要用户确认的问题。

可选 subagent：
- `explorer`：探索影响范围、现有模式、相关测试。
- `default`：在复杂需求中辅助 brainstorm 替代方案、用户体验、业务边界或架构取舍；只输出建议，不修改文件。

### 2. Test Writer
目标：先写或调整测试，尽量形成 TDD 红灯。

要求：
- 优先修改 `tests/`。
- 不修改生产代码，除非测试辅助结构必须极小调整且已说明原因。
- 写完后运行最小测试命令，确认新测试能暴露当前缺口。
- 如果无法做红灯测试，说明原因，并改用回归测试或 characterization test。
- Test Writer 阶段结束后，Codex 自行判断测试是否足以约束计划中的实现范围；足够时直接进入 Executor，不需要等待用户确认。
- 如果测试结果暴露计划错误、Verify 标准需要变化，或实现范围超出已确认计划，必须回到 Planner 更新并等待用户确认。

可选 subagent：
- `worker`：在清晰限定的测试文件中补测试。

### 3. Executor
目标：按计划实现最小必要变更。

要求：
- 只修改与需求直接相关的文件。
- 不做无关重构。
- 保持项目规范：Python 3.13+、类型提示、中文注释/文档字符串、依赖精确版本。
- 每完成一层关键变更后，按风险决定是否先运行局部测试。
- Executor 阶段结束后，Codex 自行判断实现是否已覆盖计划范围；足够时直接进入 Verify，不需要等待用户确认。
- 如果实现中发现必须改变计划、测试策略或 Verify 标准，必须回到对应阶段更新并等待用户确认。

可选 subagent：
- `worker`：仅在写入范围互不重叠时承担独立实现切片。

### 4. Verify
目标：按 planner 阶段定义的 Verify 标准证明改动可工作。

默认命令：
```bash
uv run pytest tests/ -xvs
uv run ruff check .
uv run mypy .
```

要求：
- 先执行 Planner 指定的自动化最小验证命令，再按风险执行自动化完整验证命令。
- 执行用户在 Planner 阶段给出的冒烟端到端验证，并记录执行入口、输入、输出或人工观察结果。
- 如果用户在 Planner 阶段要求全面端到端验证，则执行对应多个测试并记录结果；如果用户已明确确认省略，则记录省略确认、原因和剩余风险。
- 自动化验证、冒烟端到端验证或全面端到端验证不足以覆盖需求行为时，必须补充行为验收或针对性测试，并按阶段门禁回到 Planner 更新 Verify 标准。
- 如果命令因环境、依赖、外部服务不可用而失败，需要说明失败原因和已验证的替代证据。
- 不为了通过测试删除或弱化已有测试。
- Verify 阶段结束后，Codex 自行判断验证证据是否满足 Planner 定义的 Verify 标准；满足时直接进入 Reviewer，不需要等待用户确认。
- 如果验证失败或证据不足，进入 Fix Loop 或回到对应阶段更新计划/测试/实现。

### 5. Reviewer
目标：审查当前 diff 中的 bug、回归风险、测试缺口，并合并轻量 audit 检查。

要求：
- 不修改文件。
- 使用 `docs/checklists/review.md`，必要时参考 `docs/checklists/audit.md` 中的安全、数据、发布风险项。
- 发现问题时按严重程度排序，并引用文件和行号。
- 没有发现问题时，也要说明剩余风险或未覆盖测试。
- side project 默认不单独执行 Auditor 阶段；只有涉及安全、持久化数据、外部发布、敏感配置或用户明确要求时，才单独执行 audit。
- Reviewer 阶段结束后，必须等待用户明确同意 review 结论；如存在阻塞问题，进入 Fix Loop 前也要确认修复范围。

可选 subagent：
- `reviewer` 或 `explorer`：只读 diff 和相关上下文，输出 findings。

### 6. Fix Loop
目标：处理 review/verify 反馈。

要求：
- 明确区分必须修复、可后续优化和需要用户决策的问题。
- 由 executor 或主线程实施修复。
- 修复后重新运行相关测试和质量命令。
- 循环直到没有阻塞性问题，或需要用户决策。
- Fix Loop 中 3 轮以内的修复、验证和重新 review 可由 Codex 自行判断并继续推进，不需要每轮等待用户确认。
- 如果连续 3 轮以上修改仍未解决同一类阻塞问题，必须暂停并让用户决定是否继续、调整范围或接受风险。

### 7. Documentation Sync
目标：在所有实现、验证、review 和修复完成后，同步更新 agent 可复用的项目知识和本次计划记录。

要求：
- 检查是否需要更新 `AGENTS.md`、`docs/codemap.md` 和本次 `docs/plans/<YYYY-MM-DD>-<short-slug>.md` 计划文档。
- 本次计划文档必须更新到和最终实现一致，包括实际改动范围、最终 Verify 结果、偏离原计划的原因、未完成项和后续建议。
- 只记录对后续 agent 有价值的稳定事实，例如新模块职责、入口命令、关键数据流、测试命令、项目约束变化。
- 不记录一次性调试过程、临时计划、过期 TODO 或执行日志。
- 如果没有需要更新的内容，在最终回复中说明无需更新原因。
- 文档同步后如涉及格式或链接风险，进行轻量检查。
- Documentation Sync 阶段结束后，必须等待用户明确同意最终文档同步结果，才能输出最终完成结论。

## 最终输出
最终回复保持简短，包含：
- 改了什么。
- 验证了什么。
- 文档同步了什么，或为什么无需同步。
- 哪些命令未能运行或失败。
- 还剩哪些需要用户决策的风险。
